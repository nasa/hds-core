# Text Input

Text inputs and textareas are form elements used to capture text from users. Text inputs handle short strings (names, emails, URLs). Textareas handle longer content (messages, descriptions). Both use standard USWDS [`.usa-input`](https://designsystem.digital.gov/components/text-input/) and [`.usa-textarea`](https://designsystem.digital.gov/components/text-input/) markup with no additional classes.

## When to use text inputs

- **Short text** — names, email addresses, phone numbers, search terms, single-line responses.
- **Free-form input** — when the answer can't be predicted from a fixed set of options.
- **Longer content** — use a textarea for messages, descriptions, or multi-line input.

## When to consider something else

- **Choosing from a list** — use [radio buttons](?path=/docs/components-radio-button-guidance--docs) (fewer than 7 options) or a [select field](?path=/docs/components-select-guidance--docs) (7 or more options).
- **Multiple selections from a list** — use [checkboxes](?path=/docs/components-checkbox-guidance--docs).
- **Date input** — use a date picker for calendar dates rather than a free-text field.
- **Yes/No questions** — use radio buttons rather than a text field.

## Usability guidance

- **Always include a visible label.** Labels go above the field. Never rely on placeholder text alone — it disappears on focus and isn't announced by all screen readers.
- **Use help text for formatting guidance.** Character limits, accepted formats, and examples go in help text below the field, not in the label.
- **Match width to expected input.** A zip code field shouldn't be as wide as an address field. Use the USWDS width classes (`usa-input--sm`, `usa-input--md`, etc.) to give users a visual hint about expected length.
- **Mark required fields with an asterisk.** Combine the asterisk with a text description at the top of the form: "Required fields are marked with an asterisk (\*)."
- **Mark optional fields when most are required.** Add "(optional)" to the label text.
- **Use `inputmode` for mobile keyboards.** Set `inputmode="email"`, `inputmode="tel"`, or `inputmode="numeric"` to show the appropriate keyboard on mobile devices.
- **Don't use placeholder as a label.** Placeholder text is for examples only. It has low contrast by design and disappears when the user types.

<Note type="figma">For compact fields (e.g., a single email input in a subscription banner), HDS Figma shows the label inside the field as placeholder text that moves above on focus. This floating label pattern requires JavaScript and is deferred to a future phase.</Note>

## Accessibility

- **Every input needs a `<label>`.** The label's `for` attribute must match the input's `id`. This is required — clicking the label should focus the input.
- **Link help text with `aria-describedby`.** When help text is present, add `aria-describedby` on the input referencing the help text element's `id`. Screen readers announce the label first, then the description.
- **Link error messages with `aria-describedby`.** When both help text and an error message are present, reference both IDs: `aria-describedby="hint-id error-id"`.
- **Use `role="alert"` on error messages.** This ensures screen readers announce the error immediately when it appears.
- **Add the `required` attribute** to required fields. This provides native browser validation and announces "required" to screen readers.
- **Keyboard navigation:** Users move between fields with **Tab** and **Shift+Tab**.
- **Don't use `autocomplete="off"`** unless there is a strong security reason. Autocomplete helps users fill forms faster and reduces errors.

For general accessibility guidance, see [Accessibility](?path=/docs/foundations-accessibility--docs).
