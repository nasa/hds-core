# HDS Core Section 508 Conformance

Accessibility conformance documentation for the NASA software release process. Covers WCAG 2.0 Level A, WCAG 2.0 Level AA, Revised Section 508 Chapter 5 (Software), and Chapter 6 (Support Documentation).

Last updated: 2026-04-02

## Approach

HDS Core is a CSS/Sass theme layer on [USWDS v3.13.0](https://designsystem.digital.gov/). USWDS publishes its own accessibility conformance report: [USWDS ACR v3.11.0 (May 2025)](https://designsystem.digital.gov/about/accessibility/) using the VPAT 2.5 International Edition format, evaluating 44 components with a combination of automated testing, manual assistive technology testing, and moderated usability studies.

HDS Core inherits USWDS's conformance baseline and documents additional testing for what HDS Core changes on top of USWDS:

- **Color palette system** — 6 palettes with custom CSS properties overriding all USWDS colors. Contrast verified via automated axe-core testing across all palette and pseudo-state combinations, plus manual review of non-text elements.
- **Typography** — Custom font families (Inter, Public Sans, DM Mono), per-element line-heights and letterspacing. Resize behavior verified via 200% browser zoom.
- **Focus ring** — Two-layer architecture: USWDS theme settings plus palette-aware `:focus-visible` overrides. Visibility verified manually on all six palettes.
- **Component overrides** — CSS-only overrides on ~18 USWDS components. Markup is unmodified (except `role="list"` on ordered lists for Safari VoiceOver).
- **Net-new components** — Icon Button and Primary Arrow Button (no USWDS equivalent). Tested independently for keyboard, screen reader, and contrast conformance.

HDS Core v1.0 does not ship several USWDS components that have known accessibility issues in the USWDS ACR (step indicator, range slider, input mask, combo box, time picker, language selector, tooltip, button group, file input, character counter). Where USWDS reported "Partially Supports" due to issues in those components, HDS Core reports "Supports" because those components are not in scope. See the Component Roadmap for Phase 2 plans and the a11y implications of shipping those components in future releases.

Automated testing: Vitest 4.x runs axe-core (WCAG 2.1 A + AA) on every Storybook story in headless Chromium via Playwright. Palette-aware components have dedicated PaletteA11y stories testing contrast across all six palettes including hover and focus-visible states.

## WCAG 2.1 Level A and AA — Beyond Required Scope

The NASA software release form requires WCAG 2.0 A/AA conformance. This section documents best-practice evaluation against WCAG 2.1 criteria that are **new in 2.1** (not present in 2.0). HDS Core's automated test suite (axe-core) already runs WCAG 2.1 A+AA rules on every story — criteria with automated coverage are noted below.

This section is informational and not required for the release process.

### WCAG 2.1 Level A (new criteria)

|---|---|---|---|
| 1.3.4 | Orientation | Supports | HDS Core components do not restrict display to a single orientation. No CSS `orientation` locks are used. |
| 2.1.4 | Character Key Shortcuts | Not Applicable | HDS Core does not implement character key shortcuts. All keyboard interaction uses standard Tab, Enter, Space, Escape, and Arrow keys via USWDS JavaScript. |

## Testing Methodology

Conformance was evaluated using the following methods:

**Automated testing (continuous):**
- axe-core via `@storybook/addon-a11y` and `@storybook/addon-vitest`, running in headless Chromium via Playwright
- axe-core rule set covers WCAG 2.1 A+AA, providing automated coverage for criteria including 2.5.3 (Label in Name), 1.3.5 (Identify Input Purpose validation), and text contrast (1.4.3); criteria requiring visual inspection or interaction testing (1.4.11, 1.4.12, 1.4.10) are not covered by automated testing
- Dedicated PaletteA11y stories test contrast across all six color palettes (white, light, midtone, dark, blue, black) including hover and focus-visible pseudo-states for interactive components
- Results via `npm test` (Vitest CLI)

**Manual testing (v1.0 baseline):**
- Keyboard navigation: Tab, Shift+Tab, Enter, Space, Escape, Arrow keys on all interactive components
- Screen reader: Apple VoiceOver on macOS
- Visual inspection: Focus ring visibility across all six palettes
- Zoom: 200% browser zoom on all Storybook stories
- Form patterns: Error identification, label association, required field indication

**Non-text contrast (WCAG 2.1 — beyond required scope):**
- Visual inspection: Pseudo-element and CSS-generated content contrast (list markers, accordion chevron, table sort icons, external link arrow, link underline, checkbox icon, icon button glyphs) — axe-core cannot inspect these elements
- Ratios documented in DESIGN.md
- Icon button utility circle stroke determined to be decorative framing (see DESIGN.md § Icon Button for rationale)

**Upstream conformance:**
- USWDS v3.13.0 — see [USWDS ACR v3.11.0 (May 2025)](https://designsystem.digital.gov/about/accessibility/) for upstream evaluation covering 44 components using ANDI, pa11y, VoiceOver, NVDA, JAWS, and moderated usability studies with 13 participants

**Tools:**
- axe-core (via Storybook addon-a11y and addon-vitest)
- Vitest 4.x + Playwright (Chromium)
- Apple VoiceOver on macOS
- Browser developer tools (Chrome, Safari)

**Scope:**
- HDS Core v1.0 component inventory: Accordion, Breadcrumb, Button (CTA/Secondary/Outline/Unstyled), Checkbox, Chip (CSS-only), Form (composed), Icon Button, In-Page Navigation, Intro Text, Link, List, Pagination, Primary Arrow Button, Prose, Radio Button, Select, Site Alert, Table, Tag, Text Input
